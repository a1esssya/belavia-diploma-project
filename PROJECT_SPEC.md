# PROJECT-SPEC.md

## 1. Контекст проекта

Проект создаётся в рамках дипломной работы по теме:

**«Модернизация автоматизированной системы продажи билетов для ОАО “Авиакомпания Белавиа”».**

Практическая реализация покрывает только первую очередь модернизации.

## 2. Бизнес-цель первой очереди

Первая очередь должна:

- увеличить долю post-booking self-service операций;
- снизить нагрузку на контактный канал по типовым обращениям;
- создать архитектурную базу для следующей очереди без переработки ядра.

## 3. Scope первой очереди

### Входит

- личный кабинет;
- список заказов;
- карточка заказа;
- документы;
- повторная отправка документов;
- история событий;
- проверка доступности обмена;
- проверка доступности возврата;
- расчёт обмена до подтверждения;
- расчёт возврата до подтверждения;
- полный self-service exchange;
- полный self-service refund;
- обновление заказа, документов и истории;
- альтернативный доступ к заказу без авторизации по PNR/номеру билета и фамилии.

### Не входит

- центр уведомлений;
- SMS/push;
- управление согласиями;
- расширенные статусы;
- улучшенный календарь;
- альтернативы при отсутствии рейсов;
- мобильное приложение;
- SEO-страницы;
- переработка покупки билета;
- реальная интеграция с PSS Leonardo;
- реальная оплата.

## 4. Фиксированный стек

- Frontend: React.js SPA
- Styling: Tailwind CSS
- Routing: React Router
- Backend: NestJS
- Database: PostgreSQL
- ORM: Prisma
- Integration: mock API PSS Leonardo

## 5. Архитектура

### 5.1 Frontend

Frontend отвечает только за:

- визуальный слой;
- маршруты;
- формы;
- вызовы backend API;
- UI-состояния.

### 5.2 Backend

Backend отвечает за:

- авторизацию;
- список заказов;
- карточку заказа;
- документы;
- resend;
- историю событий;
- eligibility;
- quote;
- confirm;
- обновление витрины заказа;
- интеграцию с mock PSS.

### 5.3 Data + Integrations

PostgreSQL хранит данные цифрового канала.
Mock PSS Leonardo имитирует мастер-данные и post-booking операции.

## 9. Mock PSS Leonardo

Mock PSS должен покрывать сценарии:

- будущий заказ, доступный для обмена и возврата;
- будущий заказ с доплатой;
- заказ с запрещённым возвратом;
- отменённый заказ;
- прошедший заказ;
- ошибка поиска по PNR;
- просроченный quote;
- ошибка commit на стороне PSS.

## 11. Ключевые пользовательские сценарии

### Login

Пользователь вводит email, получает OTP, подтверждает вход и попадает в список заказов.

### Trips list

Пользователь видит будущие, прошедшие и отменённые заказы.

### Order card

Пользователь видит:

- маршрут;
- пассажира;
- статус;
- документы;
- историю событий;
- доступные действия.

### Documents

Пользователь может открыть список документов и запросить их повторную отправку на email.

### Exchange

Flow:

- eligibility;
- quote;
- confirm;
- mock payment при необходимости;
- commit;
- обновление заказа, документов и истории.

### Refund

Flow:

- eligibility;
- quote;
- confirm;
- commit;
- обновление заказа и истории.

### Booking lookup

Пользователь вводит PNR/номер билета и фамилию и получает ограниченный просмотр бронирования без входа в личный кабинет.

## 12. Бизнес-правила

- PSS является мастер-системой, но в проекте используется mock.
- Первичная покупка билета не реализуется заново.
- Доступ к заказу проверяет backend.
- После resend, exchange и refund история пополняется.
- Confirm endpoints должны поддерживать idempotency.
- Frontend не должен обращаться к mock PSS напрямую.

## 13. Режим работы с дизайном

Экранные SVG-макеты из `design/screens` используются как visual reference.

Реализация не должна быть pixel-perfect.

Разрешается:

- нормализовать отступы;
- выравнивать сетку;
- унифицировать компоненты;
- улучшать визуальную консистентность.

Цель — получить более чистый и системный интерфейс, сохранив композицию и смысл исходных экранов.

## 14. Минимальные seed-данные

Система должна подниматься с seed-данными, в которых есть:

- 1 пользователь;
- минимум 3 заказа разных типов;
- минимум 1 заказ с доступным exchange;
- минимум 1 заказ с exchange только с доплатой;
- минимум 1 заказ с недоступным refund;
- документы;
- стартовые события.

## 15. Критерии приёмки

Результат принимается, если:

- реализованы все функции первой очереди;
- список заказов и карточка заказа открываются;
- resend документов работает;
- история событий обновляется;
- exchange проходит end-to-end;
- refund проходит end-to-end;
- frontend связан только с backend API;
- есть seed-данные;
- есть понятная инструкция по запуску.

## Визуальные и UX-принципы реализации

Интерфейс первой очереди должен опираться на визуальную идентичность бренда Belavia и быть современным, чистым и консистентным.

Базовые принципы:

- доминирующий брендовый цвет: `#49559F`;
- акцентный цвет для ошибок и критических предупреждений: `#E8252D`, использовать экономно;
- фон: светлый, с большим количеством «воздуха»;
- второстепенные поверхности и разделители: мягкие светло-серые оттенки;
- минимальный размер основного текста: `16px`;
- интерфейс проектируется по 4px-сетке: `4 / 8 / 16 / 24 / 32 px`;
- реализация должна быть mobile-first;
- все интерактивные элементы должны иметь достаточно крупные touch targets;
- все формы должны иметь понятные состояния: default, focus, error, disabled;
- валидация должна быть понятной и по возможности происходить в реальном времени;
- интерфейс не должен создавать «мёртвые зоны»: вместо тупиковых экранов нужно показывать причины недоступности действия и возможные дальнейшие шаги.

Эти принципы применяются только в рамках первой очереди и не расширяют scope проекта до сценариев покупки, календаря доступных дат, поиска рейсов и branded fares.
