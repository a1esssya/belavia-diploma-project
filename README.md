# belavia-diploma-project

## Назначение проекта

Этот репозиторий содержит практическую реализацию первой очереди модернизации цифрового канала авиакомпании «Белавиа» в рамках дипломной работы по теме:

**«Модернизация автоматизированной системы продажи билетов для ОАО “Авиакомпания Белавиа”».**

Цель текущей реализации — создать web-прототип ядра post-booking self-service.

## Scope проекта

В проект входят только функции первой очереди:

- личный кабинет;
- список заказов пользователя;
- карточка заказа;
- документы по заказу;
- повторная отправка документов на email;
- история событий по заказу;
- проверка доступности обмена;
- проверка доступности возврата;
- расчёт обмена до подтверждения;
- расчёт возврата до подтверждения;
- полный self-service обмен;
- полный self-service возврат;
- обновление заказа, документов и истории после операции;

## Что не входит в проект

В проект не входят:

- центр уведомлений;
- SMS и push;
- управление согласиями;
- расширенные статусы;
- улучшенный календарь и альтернативы при отсутствии рейсов;
- мобильное приложение;
- SEO-страницы;
- переработка воронки покупки билета;
- реальная интеграция с PSS Leonardo;
- реальная интеграция с платёжным шлюзом.

## Технологический стек

Фиксированный стек:

- Frontend: React.js SPA
- Styling: Tailwind CSS
- Backend: Node.js + NestJS
- Database: PostgreSQL
- ORM: Prisma
- Integration: mock API PSS Leonardo

## Архитектурный принцип

Проект строится по трёхслойной логике:

1. Frontend — пользовательский web-интерфейс.
2. Backend — единый прикладной API и бизнес-логика.
3. PostgreSQL + mock PSS Leonardo — данные и интеграционный слой.

Frontend не должен обращаться к mock PSS напрямую.
Вся бизнес-логика exchange/refund находится только в backend.

## Основные экраны

Обязательные маршруты:

- /login
- /trips
- /orders/:orderId
- /orders/:orderId/documents
- /orders/:orderId/history
- /orders/:orderId/exchange
- /orders/:orderId/refund
- /booking-status

## Цель результата

Результат должен быть пригоден:

- для демонстрации на предзащите и защите;
- для описания в главе 3 дипломной работы;
- для генерации и доработки кода с помощью Codex.

## Definition of Done

Проект считается минимально готовым, если:

- реализованы все обязательные функции первой очереди;
- пользователь может войти и увидеть список заказов;
- карточка заказа показывает ключевые данные и доступные действия;
- работает повторная отправка документов;
- exchange и refund проходят end-to-end через backend и mock PSS;
- после операций обновляются заказ, документы и история;
- в проекте есть seed-данные;
- проект запускается локально по инструкции.
