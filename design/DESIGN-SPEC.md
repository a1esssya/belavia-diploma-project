# DESIGN-SPEC.md

## 1. Назначение файла

Этот файл задаёт правила визуальной реализации frontend-части проекта.

Он нужен для того, чтобы:

- агент не копировал SVG-макеты буквально;
- интерфейс получился более аккуратным и системным, чем исходные экраны;
- визуальная реализация соответствовала бренду Belavia;
- стили и компоненты были единообразными на всех страницах;
- проект оставался в рамках первой очереди дипломной работы.

## 2. Роль SVG-макетов

SVG-файлы в папке `design/screens` являются **визуальными референсами**, а не pixel-perfect source of truth.

Они задают:

- общую композицию экранов;
- иерархию блоков;
- смысл интерфейсных зон;
- общий визуальный характер;
- базовую структуру страниц.

SVG-макеты **не требуют точного повторения**:

- внутренних отступов;
- межэлементных расстояний;
- точных координат;
- случайных перекосов выравнивания;
- несистемных размеров;
- визуальных огрехов.

## 3. Главный принцип реализации

Frontend должен получиться:

- чище;
- ровнее;
- системнее;
- современнее;
- удобнее в использовании,
  чем исходные SVG-макеты.

Нужно сохранить:

- композицию;
- структуру экранов;
- смысл блоков;
- общий визуальный стиль.

Разрешается:

- исправлять кривые отступы;
- выравнивать сетку;
- унифицировать размеры кнопок, полей, карточек;
- улучшать визуальную консистентность;
- делать экран более аккуратным без изменения сценария.

Нельзя:

- ломать пользовательские сценарии;
- удалять важные элементы;
- менять бизнес-смысл экранов;
- превращать интерфейс в совершенно другой визуальный стиль;
- расширять scope за пределы первой очереди.

## 4. Brand Identity

Визуальная основа интерфейса должна опираться на образ Belavia как национального перевозчика.

Ключевые ассоциации:

- василёк;
- оттенки неба;
- чистота;
- спокойствие;
- доверие;
- дружелюбие;
- гостеприимство.

Интерфейс не должен выглядеть:

- тяжёлым;
- агрессивным;
- перегруженным;
- чрезмерно «технологичным»;
- визуально дешёвым;
- случайным по стилю.

Общий визуальный характер:

- минимализм;
- много свободного пространства;
- светлый фон;
- акцент на ясность и читаемость;
- аккуратная визуальная иерархия;
- спокойный премиальный тон без лишней декоративности.

## 5. Цветовая система

### 5.1 Основные цвета

- **Primary Blue:** `#49559F`
- **Alert / Error Red:** `#E8252D`

### 5.2 Фоновые и вспомогательные цвета

- основной фон: очень светлый, близкий к белому;
- вторичные поверхности: светло-серые или светло-серо-голубые;
- линии разделения: мягкий светло-серый;
- основной текст: тёмный нейтральный;
- вторичный текст: приглушённый серый;
- success-состояния: спокойный зелёный;
- warning-состояния: умеренный янтарный/оранжевый.

### 5.3 Правила использования цветов

- `#49559F` — основной брендовый цвет и базовый цвет primary actions;
- `#E8252D` — только для ошибок, alert-сообщений и критических предупреждений;
- красный нельзя использовать как обычный декоративный акцент;
- интерфейс должен оставаться преимущественно светлым;
- второстепенные цветовые акценты не должны спорить с primary blue;
- визуальный контраст должен быть достаточным для читаемости.

## 6. Типографика

### 6.1 Общие правила

- интерфейс должен быть легко читаемым на desktop и mobile;
- базовый размер основного текста — **не менее 16px**;
- визуальная иерархия заголовков должна быть очевидной;
- текст не должен выглядеть слишком мелким или плотным;
- в одном экране не должно быть чрезмерного количества разных размеров текста.

### 6.2 Характер типографики

Типографика должна быть:

- нейтральной;
- современной;
- аккуратной;
- без лишней декоративности;
- хорошо читаемой в интерфейсных сценариях.

Допустимо использовать системный sans-serif стек или современный гротеск.
Важно сохранить ощущение официального, но удобного цифрового сервиса.

### 6.3 Рекомендуемая иерархия

- Page title: 28–32px
- Section title: 20–24px
- Card heading: 18–20px
- Body text: 16px
- Secondary text: 14px
- Small meta text: 12–13px

## 7. Сетка и отступы

Использовать **4px spacing system**.

Базовые значения:

- 4px
- 8px
- 12px
- 16px
- 20px
- 24px
- 32px
- 40px

### 7.1 Правила spacing

- одинаковые компоненты должны иметь одинаковые внутренние отступы;
- расстояния между блоками должны быть логичными и повторяемыми;
- экран не должен быть «забит» элементами;
- интерфейс должен содержать достаточно воздуха;
- визуальный ритм должен быть стабильным;
- отступы не должны быть случайными.

### 7.2 Рекомендуемая логика отступов

- page padding: 24–32px на desktop;
- page padding mobile: 16px;
- gap между крупными секциями: 24–32px;
- gap между связанными блоками: 16–24px;
- внутренний padding карточек: 16–24px;
- gap между полями формы: 12–16px.

## 8. Скругления, границы и тени

### 8.1 Скругления

Скругления должны быть умеренными и единообразными.

Рекомендуемые значения:

- small radius: 8px
- medium radius: 12px
- card radius: 16px

### 8.2 Границы

- границы должны быть тонкими и мягкими;
- агрессивные тёмные обводки использовать нельзя;
- разделители должны помогать структуре, а не дробить экран.

### 8.3 Тени

Тени должны быть:

- мягкими;
- неглубокими;
- ненавязчивыми.

Карточки могут отделяться от фона:

- либо тонкой границей;
- либо мягкой тенью;
- либо комбинацией очень лёгкой тени и светлой границы.

Нельзя использовать тяжёлые, грубые или слишком контрастные тени.

## 9. Компонентные принципы

### 9.1 Общий подход

Все повторяющиеся UI-паттерны должны собираться в переиспользуемые React-компоненты.

Нельзя собирать интерфейс как набор разрозненных уникальных блоков.

### 9.2 Компоненты должны быть единообразными

Одинаковые по смыслу элементы должны иметь:

- одинаковую высоту;
- одинаковые внутренние отступы;
- одинаковую визуальную логику;
- одинаковые состояния.

## 10. Кнопки

### 10.1 Primary button

Primary CTA:

- цвет фона: `#49559F`;
- текст — контрастный;
- визуально заметная;
- умеренно скруглённая;
- не должна выглядеть тяжёлой.

Основные сценарии:

- войти;
- подтвердить;
- продолжить;
- выполнить действие;
- запросить расчёт;
- подтвердить обмен;
- подтвердить возврат.

### 10.2 Secondary button

Secondary CTA:

- светлый фон или outline;
- меньше визуальный вес, чем у primary;
- используется для второстепенных действий.

### 10.3 Tertiary / ghost button

Используется для слабых действий:

- отмена;
- возврат назад;
- вспомогательные ссылки;
- раскрытие дополнительной информации.

### 10.4 Обязательные состояния кнопок

Каждая кнопка должна поддерживать:

- default
- hover
- active
- disabled
- loading

Для длительных операций обязательно наличие loading-state.

## 11. Формы

### 11.1 Общие принципы

Формы должны быть:

- максимально понятными;
- короткими;
- визуально спокойными;
- удобными на мобильных устройствах.

### 11.2 Поля ввода

Поля должны иметь:

- понятную подпись;
- хорошую читаемость;
- заметное focus-состояние;
- видимую ошибку;
- disabled-состояние.

### 11.3 Состояния полей

Обязательные состояния:

- default
- focus
- filled
- error
- disabled

### 11.4 Ошибки и подсказки

Ошибки должны:

- быть понятными человеку;
- не быть чрезмерно техническими;
- объяснять, что не так;
- подсказывать, что делать дальше.

Допустима валидация в реальном времени, если она не раздражает пользователя.

## 12. Карточки и контейнеры

Карточки должны:

- иметь светлый фон;
- быть визуально отделены от общего фона;
- иметь ясную внутреннюю структуру;
- не содержать визуального шума.

Особенно важно это для:

- карточек заказов;
- блоков документов;
- блоков истории событий;
- блоков расчёта exchange/refund;
- статусов операции.

## 13. Статусы и бейджи

Статусы должны быть заметны, но не кричащи.

Минимальный набор статусов:

- upcoming
- past
- cancelled
- unpaid
- processing
- succeeded
- failed
- blocked / unavailable

Статус должен читаться быстро:

- через цвет;
- через текст;
- через визуальную форму бейджа.

Цвет статуса не должен быть единственным носителем смысла — текст обязателен.

## 14. Иконки

Иконки должны:

- быть простыми;
- понятными;
- единообразными по стилю;
- не быть слишком детализированными.

Использовать иконки только там, где они реально помогают сканированию интерфейса:

- документы;
- история;
- багаж;
- время;
- пассажир;
- предупреждение;
- успех;
- ошибка.

## 15. Mobile-first

Интерфейс должен проектироваться **mobile-first**.

### 15.1 Основные правила

- крупные touch targets;
- удобные зоны взаимодействия большим пальцем;
- отсутствие слишком мелких кликабельных зон;
- вертикальная логика компоновки;
- понятная иерархия без перегруза;
- достаточные расстояния между интерактивными элементами.

### 15.2 Поведение на мобильном

На мобильном:

- контент должен естественно складываться в одну колонку;
- карточки должны быть легко сканируемыми;
- формы должны оставаться короткими и понятными;
- важные CTA должны быть визуально выделены и легко доступны.

## 16. UX-принципы для первой очереди

### 16.1 Понятность действий

Пользователь должен быстро понимать:

- где он находится;
- какой это заказ;
- что можно сделать;
- что нельзя сделать;
- почему действие недоступно;
- что произойдет после подтверждения.

### 16.2 Прозрачность операций

Для post-booking сценариев обязательно явно показывать:

- статус заказа;
- доступность обмена;
- доступность возврата;
- расчёт перед подтверждением;
- статус выполнения операции;
- результат resend;
- обновление истории событий.

### 16.3 Отсутствие тупиков

Интерфейс не должен создавать «мёртвые зоны».

Если действие недоступно, нужно показать:

- причину;
- понятное сообщение;
- безопасный следующий шаг;
- альтернативное действие, если оно есть в рамках первой очереди.

### 16.4 Ошибки

Ошибки должны:

- быть визуально заметны;
- быть сформулированы человеческим языком;
- не ломать сценарий без объяснения;
- помогать завершить действие корректно.

## 17. Структура экранов

### 17.1 Login

Экран должен быть:

- простым;
- чистым;
- сфокусированным на одном действии;
- без лишней перегрузки.

### 17.2 Trips list

На экране должны легко сканироваться:

- маршрут;
- дата;
- статус;
- ключевая информация по заказу.

Карточки не должны быть перегружены мелкими деталями.

### 17.3 Order card

Это центральный экран личного кабинета.

Он должен ясно разделять:

- маршрут и основные данные;
- пассажира;
- документы;
- историю;
- доступные действия;
- блоки exchange/refund.

### 17.4 Documents

Экран документов должен быть:

- максимально простым;
- с понятными названиями документов;
- с очевидным действием resend.

### 17.5 Event history

История должна быть:

- визуально линейной;
- простой для чтения;
- с понятными датами, временем и типом события.

### 17.6 Exchange / Refund

Эти экраны должны быть максимально ясными и безопасными.

Ключевые требования:

- пошаговая логика;
- понятный quote;
- визуально отделённый блок расчёта;
- явное подтверждение;
- понятный финальный статус.

### 17.7 Booking status

Неавторизованный просмотр бронирования должен быть более ограниченным, чем личный кабинет.

Экран должен:

- показывать только допустимый объём данных;
- не создавать ощущение полноценного аккаунта;
- направлять пользователя к основному цифровому сценарию без навязчивости.

## 18. Scope guard

Следующие идеи могут использоваться только как общий UX-ориентир, но **не должны автоматически добавляться в реализацию**, если они не описаны отдельно в PROJECT-SPEC:

- интеллектуальный поиск аэропортов;
- календарь с ценами;
- branded fares;
- альтернативные маршруты;
- сценарий покупки билета;
- центр уведомлений;
- SMS / push;
- мобильное приложение;
- SEO-страницы.

Этот файл не расширяет scope проекта за пределы первой очереди.

## 19. Правила реализации в коде

- все стили — через Tailwind CSS;
- повторяющиеся паттерны — в reusable components;
- spacing, radii, shadows, colors и typography должны быть приведены к единой системе;
- нельзя хаотично задавать разные стили для одинаковых элементов;
- визуальные токены должны использоваться последовательно на всех страницах.

## 20. Источник приоритета

Приоритет источников такой:

1. `PROJECT-SPEC.md`
2. `design/DESIGN-SPEC.md`
3. `design/screens/*`
4. локальные решения агента

Если SVG противоречит бизнес-логике или scope первой очереди, приоритет у `PROJECT-SPEC.md`.

## 21. Целевой результат

На выходе нужен интерфейс, который:

- сохраняет характер бренда Belavia;
- выглядит современно и аккуратно;
- не копирует визуальные ошибки исходных SVG;
- удобен на мобильных устройствах;
- поддерживает сценарии первой очереди;
- пригоден для демонстрации и описания в дипломной работе.
